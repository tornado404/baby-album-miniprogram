# 宝宝日记核心页面产品分析报告

> **版本**: v1.0 | **最后更新**: 2026-06-17
> **状态**: 📝 设计阶段
> **配套**: [`Design.md`](./Design.md)（Claymorphism UI 规范）、[`Spec.md`](./Spec.md)（功能规格说明）

## 1. 背景与目标

本报告评估当前宝宝成长日记小程序的核心页面设计，并提出下一阶段信息架构与交互优化方向。研究范围包括：

- 首屏 / 首页：`miniprogram/pages/album_home/album_home.*`
- 上传记录：`miniprogram/pages/upload/upload.*`
- 素材库：`miniprogram/pages/gallery/gallery.*`
- 成长历程：`miniprogram/pages/journey/journey.*`
- 内容详情：`miniprogram/pages/media_detail/media_detail.*`（辅助参考）
- 底部导航：`miniprogram/components/bottom-nav/*`

目标是回答四个产品问题：

1. 一个优秀的宝宝日记产品，首屏应该是什么？
2. 是否需要素材库？如果需要，它与成长历程如何区分？
3. 上传图片 / 视频的交互应该如何设计？
4. 当前四个页面应如何重组为更清晰的产品主路径？

## 2. 当前页面现状

### 2.1 首屏 / 首页：`album_home`

`album_home` 当前承担“宝宝媒体流首页”的职责：展示当前宝宝、年龄、月份筛选、媒体卡片瀑布流和右下角上传入口。

当前信息层级：

1. 顶部标题：宝宝成长日记
2. 宝宝信息卡：头像、姓名、年龄、宝宝档案入口
3. 滚动后折叠为宝宝头像横条，支持切换宝宝 / 添加宝宝
4. 月份筛选 chips
5. 双列媒体卡片网格
6. 空状态：暂无照片，上传第一张记录
7. 右下角上传 FAB
8. 底部导航

主要问题：

- 它像首页，但底部导航中没有“首页”tab；当前 `album_home.wxml` 使用 `<bottom-nav current="journey">`，会把首页视觉标记为成长历程。
- 它展示媒体列表，但与 `journey` 的成长时间线和 `gallery` 的素材网格有职责重叠。
- 月份筛选已有 UI，但真实筛选价值还不完整。
- 首页应承担“今日记录”和“成长回看”的情感入口，而不应只是媒体列表。

### 2.2 上传记录：`upload`

`upload` 当前提供三种上传方式：拍照上传、相册选择、3D 模型。上传流程已打通：获取签名、PUT 到 MinIO、创建媒体记录、返回页面。

当前优势：

- 拍照和相册入口清晰。
- 相册选择支持多图 / 视频。
- 与后端上传链路已经打通。

主要问题：

- 页面有 `isUploading`、`uploadProgress`、`uploadStatus` 数据，但 WXML 未展示上传中、进度、失败重试。
- 上传被设计成“文件传输”，而不是“记录一个成长瞬间”。
- 缺少选择后确认：宝宝、日期、一句话描述、里程碑标记。
- 上传完成后只是 `navigateBack`，用户不一定知道内容去了哪里。
- “3D 模型”与日常记录主任务距离较远，不宜占据一线主入口。

### 2.3 素材库：`gallery`

`gallery` 当前是“素材管理工作台”：展示全部 / 未归档 / 已归档素材，支持批量选择、归档、标签、删除。

当前优势：

- 已具备批量管理框架。
- 适合处理多图上传、补标签、清理内容。
- 长按选择和批量操作方向正确。

主要问题：

- “素材库”是偏工具型概念，普通家庭用户不一定理解。
- “归档”概念不清晰：它是加入成长历程、隐藏、精选，还是完成整理？
- 当前 `journey` 空状态提示“在素材库中归档后展示在这里”，会造成“上传后还要再整理才能看到成长”的负担。
- 顶部右侧“批量选择”属于业务按钮，与项目“右上角禁止放置业务按钮”的小程序规则冲突。

### 2.4 成长历程：`journey`

`journey` 当前展示已归档内容的成长时间线，包含宝宝头部、里程碑图标横条、按月份分组的媒体卡片。

当前优势：

- “成长历程”名称贴合产品核心情感价值。
- 按月份分组有时间线基础。
- 适合作为家庭回看和分享主页面。

主要问题：

- 当前只请求 `archived=true` 内容，导致上传后的内容不会自然出现在成长历程。
- 时间线按自然月份组织，但宝宝产品更关键的是宝宝年龄阶段和里程碑。
- 里程碑横条只有图标，无状态、文案、完成时间和筛选能力。
- 顶部右侧分享按钮违反项目右上角禁区规则。

## 3. 三种信息架构方案

### 3.1 方案 A：情感时间线优先，弱化素材库

底部导航：

1. 成长
2. 相册
3. 记录
4. 我的

页面分工：

- 成长：宝宝年龄、最近记录、时间线、里程碑
- 相册：全部照片视频浏览
- 记录：上传 / 拍照 / 写文字
- 我的：宝宝档案、家人共享、设置

优点：

- 最符合家庭用户心智。
- 低复杂度，容易理解。
- 首屏情感价值强。

缺点：

- 弱化当前已有的批量整理能力。
- 如果用户一次上传大量素材，后续整理入口不够强。

### 3.2 方案 B：记录工作流优先，保留素材库为主入口

底部导航：

1. 首页
2. 素材库
3. 上传
4. 成长历程
5. 我的

页面分工：

- 首页：宝宝信息、今日记录、最近动态
- 素材库：所有素材管理
- 上传：添加素材
- 成长历程：精选归档展示
- 我的：设置

优点：

- 当前页面改动少。
- 保留重度管理能力。
- 适合多素材、多家庭成员上传后的整理工作流。

缺点：

- 主路径过长：上传 → 素材库 → 归档 → 成长历程。
- 对普通用户偏复杂。
- “素材库”和“成长历程”都看照片，职责容易混淆。

### 3.3 方案 C：成长为主线，整理为辅助（推荐）

底部导航：

1. 成长
2. 整理
3. 记录
4. 我的

页面分工：

- 成长：默认首页，展示宝宝成长时间线、最近记录、里程碑。
- 整理：原素材库，处理待补充信息、批量管理、隐藏、设为精选。
- 记录：快速上传照片 / 视频 / 文字记录。
- 我的：宝宝档案、家人共享、设置。

推荐理由：

- 符合宝宝日记的情感主线：用户最想看到的是宝宝怎么长大。
- 上传后的内容默认进入成长页，不需要用户先去素材库归档。
- 保留素材库的管理能力，但把它改成后台整理工具。
- 对新手和重度整理用户都友好。

## 4. 核心导航页面的产品设计思想

### 4.1 成长：给家人看的回忆首页

“成长”是产品的情感中心，也是默认首屏。它不应该只是图片列表，而应该回答三个问题：

1. 宝宝现在多大了？
2. 最近发生了什么？
3. 有哪些值得回看的成长瞬间？

设计原则：

- **默认可见**：上传后的有效内容默认出现在成长页，不要求先“归档”。
- **年龄优先**：宝宝日记不应只按自然月份组织，还应按“第几天 / 几个月 / 里程碑阶段”组织。
- **叙事优先**：成长页展示的是“故事”，不是素材库存。标题、日期、年龄、描述、里程碑应比管理操作更突出。
- **适合分享**：这是给家人看的页面，分享入口应放在宝宝卡、阶段卡或底部操作区，而不是右上角。

推荐模块：

1. 宝宝成长卡：头像、姓名、精确年龄、记录总数、最近记录日期。
2. 今日记录入口：拍照 / 相册 / 写一句话。
3. 最近记录：最近 3-6 条照片或视频。
4. 里程碑摘要：最近达成、待记录里程碑。
5. 成长时间线：按宝宝年龄阶段和自然月份组合展示。

### 4.2 整理：给记录者用的后台工作台

“整理”是从素材库演化而来，但不应承担主要回看职责。它的用户心智是：我来把照片和视频整理好。

设计原则：

- **任务导向**：重点不是“看”，而是“补全、筛选、隐藏、删除、设为精选”。
- **待处理优先**：默认展示“待补充”的内容，如缺描述、缺日期、未标里程碑、上传失败。
- **批量友好**：适合处理多图上传、家人上传、重复素材。
- **不制造门槛**：内容是否出现在成长页，不应依赖用户先整理；整理只是提升质量。

推荐模块：

1. 筛选：全部、待补充、已精选、已隐藏。
2. 待补充标签：缺描述、缺日期、缺里程碑。
3. 批量操作：补标签、改日期、设为精选、隐藏、删除。
4. 卡片信息：缩略图、日期、宝宝、是否已精选、缺失项提示。

命名建议：

- 推荐：“整理”
- 次选：“相册”
- 不推荐继续使用：“素材库”作为主导航名称

### 4.3 记录：最短路径完成一次成长记录

“记录”是产品的输入中心。它的目标不是上传文件，而是用最少步骤记录一个有意义的成长瞬间。

设计原则：

- **快速**：拍照和相册选择应是一屏可达。
- **轻量补充**：选择后只要求最必要信息，描述、里程碑可跳过。
- **有反馈**：上传中有进度，失败可重试，完成后明确告诉用户内容去了哪里。
- **可继续**：支持“继续记录”和“查看刚刚记录”。

推荐流程：

1. 选择来源：拍照、相册、视频、文字记录。
2. 选择后确认：预览、宝宝、日期、一句话描述、里程碑。
3. 上传中：进度、当前第几张、失败重试。
4. 完成后：已加入成长，按钮为“查看记录 / 继续上传”。

## 5. 页面级改进规格

### 5.1 成长页 / 首屏

建议将 `album_home` 与 `journey` 的职责合并或明确分工。推荐最终以“成长”为默认首页，承担当前 `journey` 的时间线能力和 `album_home` 的宝宝头部、快速记录入口。

关键改动：

- 底部导航 current 不应在首页写成 `journey` 除非首页就是成长页。
- 上传后内容默认进入成长页。
- 时间线不再只请求 `archived=true`。
- 增加年龄阶段筛选：全部、0-1月、2-3月、4-6月、7-12月、1岁+。
- 里程碑横条变为可点击筛选，并显示完成状态。
- 空状态分层：无宝宝、无记录、加载失败。

### 5.2 整理页（原素材库）

建议将 `gallery` 从“素材库”改为“整理”。

关键改动：

- 顶部标题改为“整理”。
- “批量选择”移出右上角，放在内容区或底部浮动工具条。
- “归档”改名为“设为精选”或“加入成长”。
- 默认筛选从“全部”改为“待补充”或保留全部但突出待补充数量。
- 批量标签未实现前不应作为主要按钮暴露，或需显示明确的“开发中”。
- 删除需说明影响范围：删除后会从成长页和整理页同时移除。

### 5.3 记录页（原上传页）

建议将 `upload` 从“上传记录”改为“记录”。

关键改动：

- 文案从“点击上传照片或视频”改为“今天记录宝宝的一个瞬间”。
- 上传方式改为：拍照、相册、视频、写一句话。
- 选择素材后进入确认页 / 底部弹层。
- 增加上传进度、失败重试和完成反馈。
- 无宝宝时，进入页面即引导创建宝宝档案。
- 3D 模型降级为“更多记录方式”，不作为主入口。

### 5.4 详情页

详情页应是单条成长记录的编辑和分享入口。

关键改动：

- 右上角更多按钮移到底部操作区或内容卡片内。
- 操作项：编辑、设为里程碑、分享、保存到相册、删除。
- 编辑不只改描述，应支持日期、宝宝、标签、里程碑一起编辑。
- 横图 / 竖图 / 视频需有一致的展示策略。

## 6. 用户任务路径

### 6.1 新手首次记录

目标路径：

1. 创建宝宝档案。
2. 进入成长页，看到宝宝年龄。
3. 点击“记录第一张照片”。
4. 上传完成后，成长页立刻出现第一条记录。

### 6.2 日常快速上传

目标路径：

1. 从底部“记录”进入。
2. 拍照或选相册。
3. 可选补一句话。
4. 上传完成，提示“已加入成长”。
5. 可继续上传或查看刚刚记录。

### 6.3 回看某阶段

目标路径：

1. 从“成长”进入。
2. 选择年龄阶段或里程碑。
3. 查看对应照片 / 视频 / 故事。
4. 分享阶段记录给家人。

### 6.4 整理素材

目标路径：

1. 从“整理”进入。
2. 查看待补充内容。
3. 批量补标签、改日期、设为精选或隐藏。
4. 返回成长页查看整理后的故事线。

### 6.5 分享给家人

目标路径：

1. 从成长页选择单条、阶段或宝宝主页。
2. 预览分享卡片。
3. 分享给家人。
4. 家人看到的是成长故事，不是素材管理页。

## 7. 优先级建议

### P0：明确主路径

- 底部导航改为“成长 / 整理 / 记录 / 我的”。
- 上传后默认展示在成长页。
- 成长页不再依赖 `archived=true` 才有内容。
- 明确 `album_home` 与 `journey` 的关系：合并或只保留一个作为默认成长页。

### P1：完善记录体验

- 上传进度展示。
- 上传后确认反馈。
- 选择宝宝、日期、描述、里程碑。
- 无宝宝时先引导建档。

### P2：重塑整理页

- 素材库改名整理。
- 归档改名精选 / 加入成长 / 隐藏。
- 增加待补充筛选。
- 完成批量标签和修改日期。

### P3：强化情感表达和分享

- 单条记录分享。
- 阶段分享。
- 里程碑故事卡。
- 家人共享权限。

## 8. 结论

当前产品已经具备上传、媒体展示、整理、成长时间线的基础模块，但四个页面之间职责重叠，且“成长”和“素材管理”的优先级倒置。宝宝日记的核心不是管理素材，而是让家人低成本记录、自然回看宝宝成长。

推荐采用“成长为主线，整理为辅助”的信息架构：

- **成长** 是默认首页和情感成果页。
- **记录** 是最短输入路径。
- **整理** 是后台工作台，帮助提升记录质量。
- **我的** 承载宝宝档案、家人共享和设置。

这一方向能最大化复用当前代码资产，同时把产品体验从“上传和管理图片”转向“记录和回看成长”。
