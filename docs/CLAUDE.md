# docs/ 目录规范指南

本文档约束 `docs/` 目录下的文档组织、命名、内容规范。

---

## 目录结构

```
docs/
├── CLAUDE.md                       ← 🔒 本规范文件
├── README.md                       ← 📋 文档中心导航总览（自动维护索引）
├── PRD-v2.md                       ← 🎯 顶层产品需求入口（常驻，不移动）
│
├── 01-requirements/                # 需求规格 — 为什么做 & 做什么
├── 02-design/                      # 设计规范 — 做成什么样
├── 03-architecture/                # 系统架构 — 整体怎么搭
├── 04-features/                    # 功能模块 — 具体怎么做
├── 05-testing/                     # 测试体系 — 怎么验证
├── 06-standards/                   # 开发规范 — 要遵守什么
├── 07-plans/                       # 项目计划 — 怎么做 & 做得怎样
└── archive/                        # 历史归档 — 旧版 & 废弃文档
```

### 编号分类说明

| 编号 | 目录 | 职责 | 典型读者 |
|------|------|------|----------|
| 01 | `requirements/` | 产品需求、PRD、功能规格 | 产品/设计 |
| 02 | `design/` | UI/UX 设计规范、技术选型说明 | 产品/前端 |
| 03 | `architecture/` | 系统架构、后端设计、API 定义 | 后端/全栈 |
| 04 | `features/` | 功能模块设计、用户故事 | 前端/后端 |
| 05 | `testing/` | 测试策略、E2E 指南、自动化测试 | 测试/开发 |
| 06 | `standards/` | 编码规范、踩坑手册、开发约定 | 所有开发者 |
| 07 | `plans/` | 迭代计划、任务追踪、实施路线图 | PM/开发 |

## 命名规范

### 目录命名
- 顶层目录：`NN-name/`（两位数编号 + 连字符 + 英文小写）
- 子目录：小写英文 + 下划线（如 `album_v1/`, `backend/`, `superpowers/`）

### 文件命名
- **英文文档**: `PascalCase.md` 或 `kebab-case.md`
- **中文文档**: 中文小写 + 连字符（如 `e2e-快速上手.md`）
- **编号前缀**: 功能模块类使用 `FNN-`（如 `F01-user-auth.md`）
- **故事类**: 使用 `SNN-`（如 `S01-auth-stories.md`）

### 状态标签

文档标题或表格中标注以下状态标签：

| 标签 | 含义 |
|------|------|
| ✅ | 已完成 / 已定稿 |
| 📝 | 设计阶段 / 编写中 |
| 🔄 | 评审中 / 待审核 |
| ⏸️ | 暂停 / 阻塞 |
| ❌ | 废弃 / 不再维护 |
| 📋 | 索引 / 导航 |

## 内容规范

### 文档元信息

每篇文档应在开头包含元信息块：

```markdown
# 文档标题

> **版本**: v1.0 | **最后更新**: 2026-06-06
> **状态**: 📝 设计阶段
> **配套**: `docs/xxx/xxx.md`（相关文档）
```

### 交叉引用

- 内部引用使用**相对路径**：`./01-requirements/storage-PRD.md`
- 引用时尽量附带链接说明：``[`storage-PRD.md`](./01-requirements/storage-PRD.md)（存储方案设计）``
- 避免绝对路径和外部无法访问的链接

### 文件格式

- 使用 **Markdown** 格式（`.md`）
- 表格用于结构化信息展示
- 代码块标注语言（`` ```json ``、`` ```bash ``）
- 目录结构用 ` ``` `` 包裹的 tree 格式展示
- 中英文混排时，中英文之间加空格

## 归档规则 (archive/)

满足以下任一条件应移入 `archive/`：

| 条件 | 示例 |
|------|------|
| 被新版文档完全替代 | v1.x PRD 被 PRD-v2 替代 |
| 决策已废弃 | 被否决的技术方案 |
| 历史记录 | 已完成迭代的旧计划 |
| 不再维护 | 旧版测试指南 |

**归档要求**：
1. 文件移入 `archive/` 目录
2. 在归档文件开头标注被何文档替代：`> ⏪ 已归档，由 ./NN-name/xxx.md 替代`

---

## PRD-v2.md 作为顶层入口

`docs/PRD-v2.md` 是整个文档体系的顶层入口，**永远保留在 docs/ 根目录**：

- 不移动、不改名、不归档
- 只指向其他分类目录中的文档，不替代它们
- 需要更新时同步更新 `docs/README.md` 中的索引

---

## docs/README.md 索引维护

`docs/README.md` 是文档中心的总索引页，必须保持与实际目录结构一致：

- 新增/移动/删除文档后，同步更新 README.md
- 按 8 个分类目录组织表格
- 每个表格列出来主要文档及其一句话说明
- 底部保留"角色导航指南"和最后更新日期

---

## 新增文档流程

1. 确定文档所属分类（01-07）
2. 按命名规范创建 `.md` 文件
3. 在文件头部添加元信息（版本、日期、状态、关联文档）
4. 更新 `docs/README.md` 中的索引表
5. 如果文档与 PRD-v2 相关（需求/功能/设计），同步更新 PRD-v2.md 的引用

---

## 禁止行为

- ❌ 在 docs/ 根目录新增 `.md` 文件（PRD-v2.md 除外）
- ❌ 修改 `docs/CLAUDE.md` 的约束规范（通过 PR/评审）
- ❌ 归档文件直接删除（应移入 `archive/`）
- ❌ 在文档中使用绝对路径引用（如图 `file:///`）
- ❌ 将非文档文件（图片、二进制、node_modules）放入 docs/
